## Why HTTPower?

HTTPower adds production reliability patterns on top of existing Elixir HTTP clients. Instead of writing boilerplate retry logic, circuit breakers, and rate limiters yourself, HTTPower provides them out-of-the-box with sensible defaults.

```elixir
# Add to mix.exs
{:httpower, "~> 0.20.0"},
{:finch, ">= 0.19.0"}
```

```elixir
# Make your first request
{:ok, response} = HTTPower.get("https://api.example.com/users")
IO.inspect(response.status)  # 200
IO.inspect(response.body)    # %{"users" => [...]}
```

## Clean error handling

HTTPower never raises exceptions. Every request returns a result tuple, so your application stays stable even when external services fail.

```elixir
case HTTPower.get("https://api.example.com/users") do
  {:ok, %HTTPower.Response{status: 200, body: body}} ->
    process_data(body)

  {:ok, %HTTPower.Response{status: 404}} ->
    handle_not_found()

  {:error, %HTTPower.Error{reason: :timeout}} ->
    handle_timeout()

  {:error, %HTTPower.Error{reason: :service_unavailable}} ->
    # Circuit breaker is open
    use_fallback()
end
```

## Production profiles

Pre-configured profiles encode best practices for common scenarios — no tuning required.

```elixir
# Payment gateway with deduplication and aggressive circuit breaker
payment_client = HTTPower.new(
  base_url: "https://api.stripe.com",
  profile: :payment_processing
)

# High-volume API with adaptive rate limiting
api_client = HTTPower.new(
  base_url: "https://api.github.com",
  profile: :high_volume_api
)

# Microservice mesh with cascade failure prevention
service_client = HTTPower.new(
  base_url: "https://internal-service.svc",
  profile: :microservices_mesh
)
```
