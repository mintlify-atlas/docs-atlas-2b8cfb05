This guide covers deploying HTTPower in production environments with proper configuration, monitoring, and best practices.

## Application supervision tree

HTTPower starts a supervision tree automatically when added to your application. The supervision tree manages two GenServers:

- `HTTPower.Middleware.RateLimiter` — manages rate limiting state in ETS
- `HTTPower.Middleware.CircuitBreaker` — manages circuit breaker state in ETS

### Automatic startup

When you add HTTPower to your `mix.exs`, the supervision tree starts automatically:

```elixir mix.exs
def application do
  [
    mod: {MyApp.Application, []},
    extra_applications: [:logger]
  ]
end

def deps do
  [
    {:httpower, "~> 0.5.0"}
  ]
end
```

### Supervision tree structure

```
YourApp.Supervisor
├── HTTPower.Supervisor (one_for_one strategy)
│   ├── HTTPower.Middleware.RateLimiter (GenServer with ETS)
│   └── HTTPower.Middleware.CircuitBreaker (GenServer with ETS)
└── Your other children...
```

The `one_for_one` strategy means if a GenServer crashes, only that process is restarted — not the entire HTTPower supervision tree.

### Verifying supervision

Check that HTTPower's processes are running:

```elixir
# In IEx
iex> Process.whereis(HTTPower.Middleware.RateLimiter)
#PID<0.234.0>

iex> Process.whereis(HTTPower.Middleware.CircuitBreaker)
#PID<0.235.0>

# Check supervisor
iex> Supervisor.which_children(HTTPower.Supervisor)
[
  {HTTPower.Middleware.CircuitBreaker, #PID<0.235.0>, :worker, [HTTPower.Middleware.CircuitBreaker]},
  {HTTPower.Middleware.RateLimiter, #PID<0.234.0>, :worker, [HTTPower.Middleware.RateLimiter]}
]
```

## Production configuration

config :httpower,

    ]
  ]

### Per-service configuration

For applications calling multiple external services:

```elixir config/prod.exs
config :myapp, :stripe_client,
  circuit_breaker: [
    failure_threshold: 3,
    timeout: 30_000
  ],
  rate_limit: [
    requests: 100,
    per: :second
  ]

config :myapp, :github_client,
  circuit_breaker: [
    failure_threshold: 5,
    timeout: 60_000
  ],
  rate_limit: [
    requests: 60,
    per: :minute
  ]
```

```elixir
defmodule MyApp.StripeClient do
  def client do
    config = Application.get_env(:myapp, :stripe_client, [])

    HTTPower.new([
      base_url: "https://api.stripe.com",
      circuit_breaker_key: "stripe"
    ] ++ config)
  end
end
```

## Monitoring

### Circuit breaker state

Monitor circuit breaker state changes in production:

```elixir
defmodule MyApp.CircuitBreakerMonitor do
  require Logger

  def check_circuits do
    circuits = ["payment_api", "user_service", "order_service"]

    Enum.each(circuits, fn circuit_key ->
      state = HTTPower.Middleware.CircuitBreaker.get_state(circuit_key)

      case state do
        :open      -> Logger.warning("Circuit OPEN: #{circuit_key}")
        :half_open -> Logger.info("Circuit HALF-OPEN: #{circuit_key}")
        :closed    -> :ok
        nil        -> :ok  # Circuit not initialized yet
      end
    end)
  end
end

# Schedule with Quantum, Oban, or a GenServer
```

### Rate limiter metrics

Track rate limiting hits:

```elixir
defmodule MyApp.RateLimitMonitor do
  def track_rate_limit_result(result) do
    case result do
      {:error, %{reason: :too_many_requests}} ->
        :telemetry.execute(
          [:myapp, :http, :rate_limit_exceeded],
          %{count: 1},
          %{}
        )
      _ ->
        :ok
    end
  end
end

# Wrap your API calls
def fetch_data do
  result = HTTPower.get(client(), "/data")
  MyApp.RateLimitMonitor.track_rate_limit_result(result)
  result
end
```

### Request duration tracking

HTTPower logs request duration automatically. Duration also appears in telemetry and structured log metadata:

```
# Log format
[info] [HTTPower] [req_abc123] ← 200 (1234ms)

# Structured metadata fields (queryable in Datadog, Splunk, ELK)
httpower_duration_ms:>1000 AND httpower_event:response
httpower_status:>=500 AND httpower_method:post
httpower_correlation_id:"req_abc123"
```

## Performance tuning

### ETS table sizing

HTTPower uses ETS for rate limiter and circuit breaker state. For high-volume applications:

```elixir
# ETS tables are created with these defaults:
# - :set type (key-value storage)
# - :public read_concurrency
# - Named tables

# No tuning needed for most applications
# ETS handles millions of operations per second

# Check current ETS usage
:ets.info(:httpower_rate_limiter)
:ets.info(:httpower_circuit_breaker)
```

<Tip>
Memory usage for the ETS tables should be minimal — well under 10 MB in typical deployments. The rate limiter automatically cleans up inactive buckets after 5 minutes. The circuit breaker tracks only the last `window_size` requests.
</Tip>

### Connection pooling

Connection pooling is handled by the underlying adapter.

<Tabs>
  <Tab title="Req (default)">
Finch manages connection pooling automatically. Configure if needed:

```elixir config/config.exs
config :req, finch: [
  pools: %{
    default: [size: 25, count: 5]
  }
]
```
  </Tab>
  <Tab title="Tesla">
Configure your Tesla adapter's connection pool (Finch, Hackney, etc.) according to that adapter's documentation.
  </Tab>
</Tabs>

### Timeout strategy

Set appropriate timeouts (in seconds) for your use case:

```elixir
# Fast APIs
config :httpower, timeout: 10

# Slow APIs or large payloads
config :httpower, timeout: 120

# Per-request override for specific slow endpoints
HTTPower.get(client(), "/slow-endpoint", timeout: 300)
```

## High availability setup

### Multi-node considerations

HTTPower's rate limiter and circuit breaker are **per-node** (stored in local ETS). In a multi-node setup:

| Feature | Behavior | Implication |
|---|---|---|
| Rate limiting | Each node has its own buckets | Total throughput = `requests_per_node × num_nodes` |
| Circuit breaker | Each node has independent state | Circuit opens independently per node (node-level isolation) |

**Example:** `100 req/min × 3 nodes = 300 req/min` total throughput.

<Tip>
Per-node circuit breakers are usually the desired behavior — a failing node isolates its own circuits without affecting the rest of the cluster.
</Tip>

### Centralized rate limiting (advanced)

For strict global rate limits across multiple nodes, implement a centralized rate limiter using Redis or a distributed state library. HTTPower's built-in rate limiter is designed for per-node limits.

### Load balancer configuration

When using HTTPower behind a load balancer:

```elixir
config :httpower,
  timeout: 30,

  # Circuit breaker per-node is fine
  circuit_breaker: [enabled: true],

  # Set per-node limit; total is limit × node_count
  rate_limit: [
    enabled: true,
    requests: 50  # If 2 nodes, total is ~100 req/min
  ]
```

## Security best practices

### SSL verification

<Warning>
Always enable SSL verification in production. Never set `ssl_verify: false` in a production environment.
</Warning>

```elixir
config :httpower, ssl_verify: true  # Default
```

### Sensitive data logging

Configure comprehensive sanitization and structured logging for production observability:

```elixir
config :httpower,
  logging: [
    level: :info,
    log_headers: true,
    log_body: true,
    sanitize_headers: [
      "authorization", "api-key", "x-api-key",
      "cookie", "set-cookie", "x-auth-token"
    ],
    sanitize_body_fields: [
      # Auth
      "password", "secret", "token", "api_key",
      # Financial
      "credit_card", "card_number", "cvv", "cvc",
      "account_number", "routing_number",
      # Personal
      "ssn", "tax_id", "drivers_license"
    ]
  ]
```

All logs include structured metadata accessible via `Logger.metadata()`, enabling powerful queries in production log systems (Datadog, Splunk, ELK):

```
# Find slow requests in production
httpower_duration_ms:>1000 AND httpower_event:response

# Alert on error rates
httpower_status:>=500 AND httpower_method:post

# Trace customer requests
httpower_correlation_id:"req_abc123"
```

Metadata fields: `httpower_correlation_id`, `httpower_event`, `httpower_method`, `httpower_url`, `httpower_status`, `httpower_duration_ms`, and more.

### API keys

Store API keys securely using environment variables or `runtime.exs`:

```elixir
# Use environment variables
api_key = System.fetch_env!("STRIPE_API_KEY")

client = HTTPower.new(
  headers: %{"authorization" => "Bearer #{api_key}"}
)
```

```elixir config/runtime.exs
config :myapp, :stripe_api_key, System.fetch_env!("STRIPE_API_KEY")
```

### Test mode in production

<Warning>
Never enable test mode in production. Test mode bypasses real HTTP requests entirely.
</Warning>

```elixir config/prod.exs
config :httpower, test_mode: false  # Default
```

```elixir config/test.exs
# Only in test environments
config :httpower, test_mode: true
```

## Graceful degradation

Handle circuit breaker opens and rate limit errors gracefully in your application code:

```elixir
defmodule MyApp.PaymentService do
  def charge_customer(amount) do
    case HTTPower.post(payment_client(), "/charge", body: amount) do
      {:ok, response} ->
        {:ok, response}

      {:error, %{reason: :service_unavailable}} ->
        # Circuit is open - payment service is down
        Logger.error("Payment service unavailable (circuit open)")
        {:error, :service_unavailable}

      {:error, %{reason: :too_many_requests}} ->
        # Hit rate limit
        Logger.warning("Payment service rate limited")
        {:error, :rate_limited}

      {:error, error} ->
        {:error, error}
    end
  end

  # Fallback method
  def charge_customer_fallback(amount) do
    # Use secondary payment processor
    # Queue for later processing
    # Return friendly error to user
  end
end
```

## Deployment checklist

<Steps>
  <Step title="Configure circuit breaker thresholds">
    Set `failure_threshold`, `window_size`, and `timeout` values based on your SLAs and the typical error rates of each external service.
  </Step>
  <Step title="Set appropriate rate limits">
    Configure per-service rate limits to stay within vendor API quotas. Remember that limits are per-node in multi-node deployments.
  </Step>
  <Step title="Configure PCI-compliant logging">
    Add all sensitive field names to `sanitize_body_fields` and add sensitive headers to `sanitize_headers`. Set `level: :info` in production.
  </Step>
  <Step title="Set reasonable retry limits and timeouts">
    Configure `max_retries`, `base_delay`, `max_delay`, and `timeout` to match your service's latency profile.
  </Step>
  <Step title="Disable test mode">
    Verify `config :httpower, test_mode: false` (or no explicit setting) in `prod.exs`.
  </Step>
  <Step title="Set up circuit breaker monitoring">
    Schedule a periodic check of circuit breaker states and configure alerts for `:open` transitions.
  </Step>
  <Step title="Document circuit breaker keys">
    Record which endpoints use which `circuit_breaker_key` values so your on-call team can identify which service is affected when a circuit opens.
  </Step>
  <Step title="Configure environment variables">
    Store all API keys and secrets in environment variables. Use `System.fetch_env!/1` in `config/runtime.exs` to fail fast at startup if a required variable is missing.
  </Step>
  <Step title="Test fallback behavior">
    Verify your application responds gracefully when `:service_unavailable` and `:too_many_requests` errors are returned.
  </Step>
  <Step title="Verify SSL verification is enabled">
    Confirm `ssl_verify: true` (the default) is not overridden anywhere in your production config.
  </Step>
</Steps>

## Troubleshooting production issues

### Circuit keeps opening

**Debug:**

```elixir
# Check current state
HTTPower.Middleware.CircuitBreaker.get_state("my_service")

# Temporarily disable to test
config :httpower, circuit_breaker: [enabled: false]

# Check failure patterns in logs
# Look for: [HTTPower] [req_*] ← 500 or ← timeout
```

**Solutions:**
- Increase `failure_threshold` if the service has transient issues
- Increase `window_size` to smooth out spikes
- Reduce `timeout` if the service is responding slowly
- Check whether the external service is actually down

### Rate limiting too aggressive

**Debug:**

```elixir
# Check rate limit config
Application.get_env(:httpower, :rate_limit)

# Test without rate limiting
config :httpower, rate_limit: [enabled: false]
```

**Solutions:**
- Increase `requests` or change `per` to a larger window
- Use `:error` strategy instead of `:wait`
- Use a custom `rate_limit_key` to separate endpoints
- Implement per-user rate limiting in your application layer

### High memory usage

**Check ETS tables:**

```elixir
:ets.info(:httpower_rate_limiter)
:ets.info(:httpower_circuit_breaker)
```

**Solutions:**
- The rate limiter automatically cleans up inactive buckets (5 min)
- The circuit breaker tracks only the last `window_size` requests
- Memory usage should be minimal (under 10 MB typically)

## Getting help

For production issues:

1. Check logs for `[HTTPower]` messages with correlation IDs
2. Review circuit breaker and rate limit state via `get_state/1`
3. Check the [Configuration Reference](/configuration/overview) for all options
4. Open an issue with logs and config at https://github.com/mdepolli/httpower/issues
